核心挑战：为什么不仅是 "AI Gateway"?

许多企业误以为仅仅接入一个 Mendix AI Gateway 就能解决问题。实际上，网关只是通道。要真正消除幻觉，必须构建完整的RAG链路：

  • ● 知识工程： 对非结构化工业文档（手册、图纸）的精准解析。
  • ● 检索策略： 混合检索（关键词+向量）与重排序（Rerank）。
  • ● 应用编排： 利用 Mendix GenAI Commons 组件串联业务逻辑。

中国市场 RAG 技术生态全景

在中国构建企业级RAG，必须考虑国产化适配、数据合规及特定技术栈的优势。以下是当前中国市场主流的RAG组件方案。

核心编排

Ragflow

一款基于深度文档理解（DeepDoc）的开源RAG引擎。

为什么重要？

相比LangChain的通用性，Ragflow更专注于复杂文档解析（如表格、多栏排版），这对于工业手册至关重要。

向量存储

Milvus / Zilliz

全球领先的开源向量数据库，源自中国团队。

优势：

高性能、分布式架构，适合海量工业知识库的存储与毫秒级检索。在信创环境支持良好。

图谱增强

NebulaGraph / Neo4j

Graph RAG 是解决复杂推理的关键。

工业应用：

将设备（实体）与故障现象（关系）构建成知识图谱，弥补纯向量检索在逻辑跳跃上的不足。

方案优劣势深度对比

企业选择 RAG 方案时，需权衡开发速度、定制化能力与运维成本。本节对比 Mendix GenAI 方案、开源成熟方案 (如 Ragflow) 与 商业化成品 (如 百度千帆/阿里百炼)。

Mendix 方案 (PaaS/Low-Code)

  • ✔ 优势： 与现有业务系统（ERP/MES）集成极快；UI开发成本极低；利用 GenAI Commons 快速封装。
  • ✘ 不足： 对底层向量库和切分算法的控制力不如纯代码灵活；依赖组件更新。

开源方案 (Ragflow/LangChain)

  • ✔ 优势： 极致的定制化；最新的算法（如Raptor, Self-RAG）跟进最快；无License费用。
  • ✘ 不足： 运维复杂度高；UI/交互层开发工作量大；企业级权限管理需自研。

全模态数据解析与 ETL 方案

工业知识不仅存在于文档中，还大量存在于图纸、维修视频和3D模型中。高质量的 RAG 始于高质量的解析。

非结构化文档深度解析

工业手册通常包含复杂的多级标题、表格和跨页图表。简单的文本提取会导致上下文丢失。

开源方案

  • IBM Docling: 架构优选 IBM 开源的高性能解析引擎，极擅长处理工业手册中的嵌套表格、公式与跨页排版，高保真输出 LLM 友好的 Markdown/JSON，是高质量 ETL 管道的核心利器。
  • Ragflow (DeepDoc): 专为RAG设计的视觉布局分析模型，配合自身引擎体验佳。
  • Unstructured.io: 通用解析生态库，支持的长尾文件格式极多。
  • PyMuPDF / PDFPlumber: 基础底层库，需大量编写 Regex 正则与边界规则进行二次开发。

中国商业方案

  • 合合信息 (TextIn): 业界领先的文档解析API，对中文表格还原度极高。
  • 百度文心千帆 (ETL组件): 集成在平台内，开箱即用。

工业OCR与视觉理解

针对铭牌、仪表读数、扫描件图纸的文字提取。

开源方案

  • PaddleOCR (百度): 轻量级、支持80+语言，中文识别效果SOTA。
  • RapidOCR: 基于ONNXRuntime的推理库，部署极快。

多模态大模型 (VLM)

相比传统OCR，直接使用 Qwen-VL (阿里) 或 Yi-VL (零一万物) 可以直接"理解"图片内容并生成描述，存入向量库。

音视频数据解析

处理维修录音、会议记录及培训视频。

  • FunASR (阿里达摩院) 开源/中文极佳 相比Whisper，FunASR在中文工业术语识别上更具优势。
  • Video-Llama / ChatVideo 视频理解 提取关键帧并生成时序描述，用于故障检索。

3D CAD 模型与点云解析

最具挑战性的领域。目标是将3D几何特征转化为文本描述或向量。

难点提示： 目前尚无完美的开源通用3D转文本方案。主流做法是解析元数据（BOM表）或多视图截屏后使用VLM。

开源技术路径

  • PythonOCC / OpenCascade: 用于解析STEP/IGES文件的几何结构和元数据。
  • PointNet++ / ULIP: 针对点云数据的特征提取（学术界较多，工业落地较少）。

Mendix 结合方案

利用 Mendix 3D Viewer 组件展示模型，并在侧边栏通过 RAG 检索该模型关联的文档（维修手册、参数表），而非直接检索几何体。这是目前最务实的路径。

步骤 1: 知识库搭建 (Backend)

不要尝试在 Mendix 内部做复杂的 PDF 解析。使用 Ragflow 或 LangChain 在后端构建 API 服务。
关键动作： 部署 Milvus 和 Embedding 模型（推荐 BGE-M3）。

步骤 2: Mendix 集成 (GenAI Commons)

导入 GenAI Commons 模块。配置 "Request Handling" 微流，调用后端 RAG 检索接口，而不是直接调 LLM。
关键动作： 将检索到的 Context 注入 Prompt 模板。

步骤 3: 业务上下文融合

Mendix 的优势在于拥有业务数据（如设备ID、当前工单状态）。
关键动作： 在 Prompt 中加入 Mendix 对象数据："当前设备型号是 [Module.DeviceType]，故障代码 [Code]，请根据知识库回答..."

步骤 4: 人机回环 (HITL)

利用 Mendix 快速构建反馈界面。
关键动作： 允许专家在 UI 上对 AI 回答点赞/点踩，并将修正后的答案回写到知识库，形成闭环。
